# BlocRoc — Substrate Blockchain Ticketing Platform

## Project Overview

BlocRoc is a decentralized ticketing platform built on a custom Substrate blockchain. It enables event organizers to mint NFT-style tickets, fans to purchase and transfer tickets trustlessly, and venue staff to validate entry via a mobile scanner app.

### Architecture

```
BlocRoc Monorepo
├── roc-chain/          Substrate node + FRAME runtime + custom pallets
├── roc-frontend/       Next.js web app (venue dashboard + fan marketplace)
├── roc-scanner/        React Native mobile scanner (entry validation)
├── roc-indexer/        SubQuery indexer (off-chain query layer)
├── docs/               Architecture docs, API specs, ADRs
└── scripts/            Dev setup, demo, and CI helper scripts
```

### On-Chain Components (roc-chain)

| Crate | Purpose |
|---|---|
| `pallet-ticket` | Core ticket lifecycle: mint, transfer, burn, invalidate |
| `pallet-event` | Event creation, capacity management, metadata |
| `pallet-marketplace` | Secondary market: list, bid, buy, cancel |
| `pallet-scanner` | Entry validation: scan, revoke, seat assignment |
| `runtime` | WASM runtime composing all pallets via FRAME |
| `node` | P2P node, RPC server, block production |

## Rust Coding Conventions

### Error Handling
- **Never use `unwrap()` or `expect()` in production code.** Use `?` propagation or explicit `match`/`if let`.
- Define a crate-level `Error` enum using `#[pallet::error]` for pallet errors.
- Return `DispatchResult` / `DispatchResultWithPostInfo` from dispatchable calls.
- Off-chain (node, indexer) code must use `anyhow::Result` or a typed error enum — no `unwrap`.

```rust
// Bad
let account = accounts.get(&id).unwrap();

// Good
let account = accounts.get(&id).ok_or(Error::<T>::AccountNotFound)?;
```

### Types
- Prefer newtype wrappers over raw primitives for domain types (e.g., `EventId(u64)` not `u64`).
- Use `BoundedVec<T, S>` for all on-chain collections — never unbounded `Vec`.
- Use `T::AccountId`, `T::BlockNumber`, `BalanceOf<T>` — never hardcode concrete types in pallets.

### Naming
- Types: `UpperCamelCase`
- Functions/variables: `snake_case`
- Constants: `SCREAMING_SNAKE_CASE`
- Pallet storage items: `UpperCamelCase` with descriptive names (e.g., `EventById`, `TicketOwner`)

## Substrate / FRAME Conventions

### Pallet Structure
Every pallet must follow the standard FRAME layout:

```rust
#[frame_support::pallet]
pub mod pallet {
    use frame_support::pallet_prelude::*;
    use frame_system::pallet_prelude::*;

    #[pallet::pallet]
    pub struct Pallet<T>(_);

    #[pallet::config]        // Trait bounds and associated types
    #[pallet::storage]       // On-chain state declarations
    #[pallet::event]         // Events emitted by dispatchables
    #[pallet::error]         // Error variants
    #[pallet::call]          // Dispatchable extrinsics
    #[pallet::hooks]         // on_initialize, on_finalize, etc.
    #[pallet::genesis_config] // Optional genesis state
}
```

### Weights
- Every dispatchable **must** have a `#[pallet::weight(...)]` annotation.
- During development, use `Weight::from_parts(10_000, 0)` as a placeholder.
- Before mainnet, run `frame-benchmarking` and replace all placeholder weights.
- Weight functions live in `weights.rs` generated by the benchmark CLI.

### Storage
- Prefer `StorageMap` over `StorageDoubleMap` unless a compound key is truly necessary.
- Always provide explicit `ValueQuery` or `OptionQuery` — never rely on defaults.
- Document each storage item with a `///` doc comment explaining invariants.

### Events
- Emit an event for every state-changing dispatchable.
- Include enough data in the event for off-chain indexers to reconstruct state without querying storage.

### Genesis Config
- All pallets with initial state must implement `GenesisBuild<T>`.
- Keep genesis config minimal — do not duplicate data already derivable at runtime.

## Git Conventions

### Commit Messages (Conventional Commits)
```
<type>(<scope>): <short description>

[optional body]

[optional footer]
```

**Types:** `feat`, `fix`, `chore`, `docs`, `test`, `refactor`, `perf`, `ci`

**Scopes:** `chain`, `runtime`, `ticket`, `event`, `marketplace`, `scanner-pallet`, `frontend`, `scanner-app`, `indexer`, `scripts`, `docs`

Examples:
```
feat(ticket): add batch mint dispatchable
fix(marketplace): prevent double-spend on concurrent bids
test(event): add capacity overflow unit tests
docs(api): update ticket RPC spec
```

### Branch Naming
```
<type>/<scope>-<short-description>
feat/ticket-batch-mint
fix/marketplace-double-spend
chore/update-substrate-deps
```

### Rules
- `main` is always deployable — no direct pushes, PRs only.
- Feature branches cut from `main`, merged via PR after review + CI green.
- Squash-merge feature branches to keep `main` history linear.
- Tag releases as `v0.1.0`, `v0.2.0` etc. following semver.

## Testing Requirements

### Pallets (roc-chain/pallets/*)
- **Unit tests required for all pallet logic** — every dispatchable, every storage transition.
- Use `mock.rs` with a minimal `construct_runtime!` for isolated pallet testing.
- Use `frame_support::assert_ok!` / `assert_noop!` / `assert_err!` macros.
- Test both the happy path and all error branches.
- Storage state must be verified after each operation — do not only check return values.

```
src/
  lib.rs        pallet implementation
  mock.rs       test runtime
  tests.rs      unit + integration tests
  benchmarking.rs  benchmark extrinsics
  weights.rs    generated weight file
```

### Runtime (roc-chain/runtime)
- Integration tests validating cross-pallet interactions go in `runtime/tests/`.
- Use `sp_io::TestExternalities` for all runtime tests.

### Frontend / Scanner App
- Unit tests for all utility functions and hooks (Jest / React Testing Library).
- E2E tests for critical user flows (Playwright for frontend).

### Indexer
- Unit tests for mapping functions with mock SubQuery data.

## Development Setup

See `scripts/setup.sh` for automated environment setup.

Quick start:
```bash
# Install Rust + targets
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
rustup target add wasm32-unknown-unknown

# Build the chain
cd roc-chain && cargo build --release

# Start local node
./target/release/roc-node --dev

# Start frontend
cd roc-frontend && npm install && npm run dev

# Start indexer
cd roc-indexer && npm install && npm run codegen && npm run build && npm run start:docker
```
